## 2. The Recommendation

**PRIMARY: MeetingBaaS.** Honest reason: it is the only candidate that is simultaneously EU by construction (French SAS, Scaleway Paris hosting, Gladia France STT, published DPA with SCCs and sub-processors), Recall-shaped enough to swap in 1-3 days, and ~$19/month with a $0 subscription tier at Forj volume. It wins because it converts the GDPR gate from a caveat into a selling point: Forj can tell Swedish tenants the entire meeting chain (vendor, hosting, transcription) is EU-owned. Skribby is $3-6/month cheaper but is a two-founder Belgian side-project with no certifications; that vendor risk is not worth saving $5. Recall EU is one day easier but keeps a US entity in the chain, which is exactly what a Swedish tenant procurement review will poke at.

**FALLBACK: Recall.ai eu-central-1 region.** Honest reason: it is the zero-integration-risk escape hatch. If the MeetingBaaS pilot fails on Swedish quality, bot reliability, or the hosting-wording question, a fresh EU-region Recall account is one day of rewiring on an API the pipeline was literally built for, at the same ~$20/month, with EU transcription processing now confirmed in their docs. It loses only on US-entity ownership; if the gate turns out to be "EU processing" rather than "EU-owned vendor", it is fully sufficient.

Do not self-host now. The AWS-native path is the best story but costs 2-3 founder-weeks plus a permanent breakage tax during the ACTIVATION_30D window. Keep it as the documented exit ramp if a tenant ever demands zero-third-party processing.

## 3. Integration Sketch (MeetingBaaS vs existing pipeline)

**meeting-start (creates bot). CHANGES: request mapping only.**
- Replace Recall create-bot call with `POST https://api.meetingbaas.com/bots/` (v1, the stable documented shape), auth via their API key header.
- Body: `meeting_url` (same value as today), `bot_name` ("Smith, Forj notetaker"), `webhook_url` (existing meeting-webhook endpoint), `entry_message` (Swedish/English recording notice posted to meeting chat on join, this is the consent hook), `recording_mode: "speaker_view"`, `speech_to_text` (Gladia default, language sv or auto), `extra: { tenant_id, engagement_id }` (their custom-metadata passthrough, replaces whatever Recall metadata field is used today), `automatic_leave` config.
- Store returned `bot_id` where the Recall bot id lives today. STAYS: tenant caps, scheduling logic, everything upstream.

**meeting-webhook (transcript-ready). CHANGES: event names + one mandatory persist step.**
- Handle their `complete` event: payload carries `bot_id`, `meeting_url`, `recording_url` (MP4), `transcript_url`, `duration`, `participants`. Correlate via bot_id or `extra` metadata.
- MANDATORY: download transcript + MP4 into own storage (Supabase/S3) immediately in the handler. Free tier retains only 3 days. This is already the pipeline design; make it non-optional in code.
- Map Gladia speaker-separated segments onto the existing transcript schema (small mapper, speaker labels + timestamps are included at the base 1.00 token/hr tier).
- Also handle their failure event for the bot-could-not-join path.

**meeting-poll (status). CHANGES: repoint URL.**
- Point at their bot-status endpoint (GET on the bot resource). Keep as the safety net for missed webhooks, same as today.

**STAYS UNTOUCHED:** everything downstream (Smith summarization, storage schema after the mapper, tenant capping, the Recall-shaped internal interface, which remains the abstraction that makes the fallback swap a one-day job).

**Setup tasks:** create account on the $0 Pay-as-You-Go plan, buy one $50 Boost pack (100 tokens = ~80 recorded hours at 1.25 tokens/hr), 8 free hours cover the e2e test across Zoom + Teams + Meet.

**Cost model:** 1.25 tokens/hr x $0.50 = $0.625/bot-hour incl. diarized Swedish transcript. 30 hrs/month = $18.75. Tokens never expire. Trip-wires: >75 bots/day or >2 calendar integrations forces the $99/mo Pro tier; irrelevant at current volume.

## 4. Open Questions for Jacob Before Committing

1. **What is the actual gate: EU processing or EU-owned vendor?** This is THE fork. If a Swedish tenant will accept Frankfurt processing by a US company, Recall eu-central-1 is one day of work and no new vendor relationship. If the EU data moat pitch requires EU ownership, MeetingBaaS. Decide this before writing any code.
2. **One email to hello@spoke.app:** their DPA says Scaleway (Paris), their privacy policy says AWS (EU). Get written confirmation of the actual hosting provider and region before relying on the DPA. Both answers keep you in the EU, but the paper must agree.
3. **Consent bar:** MeetingBaaS gives a named bot + chat entry_message, no spoken announcement (same gap as nearly every candidate). Is chat notice + Forj's own booking-flow consent sufficient for Swedish tenants, or does the meeting-start flow need to require host acknowledgment first?
4. **Swedish quality pilot:** run 2-3 real Swedish partner calls through Gladia default on the 8 free hours. If quality misses, the fallback inside the same vendor is audio_only at 1.00 token/hr + Amazon Transcribe batch sv-SE in eu-north-1 ($0.36/hr) with the diarization timestamps MeetingBaaS already provides. Decide the quality bar before the pilot, not after.
5. **Self-host ops appetite:** recommendation assumes zero appetite during ACTIVATION_30D. Confirm, and park Attendee-on-Stockholm as the written escape hatch only.
6. **Budget ceiling:** confirm ~$20/month steady state plus a $50 prepay is approved, and that the $99/mo Pro tier trigger (75 bots/day, calendar sync) is acceptable if volume 10x's.
7. **Retention verification:** e2e test must prove transcript + MP4 land in own storage on the complete webhook, since the 3-day window makes a silent webhook failure a data-loss event. Ensure meeting-poll backstops this.
8. **Incumbent cleanup:** export any transcripts still sitting in the US Recall account and close it, so the sub-processor page can drop the US vendor the same week the swap lands.
